# AmplifyKS — Codebase Key Findings

_Last updated: 2026-07-03 (branch `main`, HEAD 4cc31e7)_

## What it is
Kansas civic-engagement app: browse KS Legislature bills, look up legislators/committees, submit testimony, take an issue quiz that matches users to legislators, and earn gamification points. Cross-platform Expo app (iOS/Android/web) deployed to Netlify as a static web export.

## Stack
- **Expo SDK 54** / React Native 0.81 / React 19, **expo-router v6** (file-based routing, typed routes + React Compiler experiments enabled in `app.json`)
- **Firebase v12**: Auth (email/password) + Firestore. Web uses `getAuth`, native uses `initializeAuth` with AsyncStorage persistence — lazy singletons in `services/firebase.ts`
- **Netlify**: `npx expo export --platform web` → `dist/`, SPA redirect in `netlify.toml`. `netlify/functions/legiscan.mts` proxies LegiScan so the API key stays server-side (`LEGISCAN_API_KEY` env var); it also rate-limits (200 req/10s per IP, in-memory) since the URL is public once deployed — best-effort only, see gotcha #7
- **Tests**: jest-expo (`npm test`), suites in `__tests__/` dirs covering the pure logic (match engine, tagger, gamification, legislator adapter, LegiScan helpers). Tests that transitively import firebase must `jest.mock('firebase/firestore', ...)` — firebase's mixed CJS/ESM dist breaks jest otherwise
- **CI**: `.github/workflows/ci.yml` runs tsc, lint, tests, and `validate:data` on pushes to main and PRs
- Shared AsyncStorage cache helpers live in `services/persistent-cache.ts` (`readPersistentCache` = stale-while-revalidate for OpenStates; `readFreshPersistentCache` = evict-on-expiry for LegiScan)

## Running it
- `npm start` / `npm run web` for normal dev
- `npm run web:netlify` → http://localhost:8888 when Netlify function proxying is needed (runs Expo web on 8082 behind Netlify dev on 8888; CI mode = **no hot reload**, manual browser refresh). The script `rm -rf dist` first — a stale `dist/` makes netlify dev's SPA redirect hijack every request (including the JS bundle) and the app renders an empty page
- Secrets in `.env` (gitignored): `EXPO_PUBLIC_FIREBASE_*`, `EXPO_PUBLIC_OPENSTATES_API_KEY`, `EXPO_PUBLIC_BILLTRACK50_API_KEY`

## Routing / app structure (`app/`)
- `index.tsx`: auth gate — logged in → `/(tabs)/dashboard`; web anon → `/lookup`; native anon → `/(auth)/login`
- Route groups: `(auth)` (login/register), `(tabs)` (dashboard, bills, actions, points, organizations, committees, profile, `officials/` sub-stack: index, state, federal, lookup). `organizations` is `href: null` like `points` (2026-07-04) — the screen is a "Coming Soon" placeholder with no content, hidden from both the mobile tab bar and desktop `WebTopNav` until the feature ships; re-add its nav entries when it does
- `committees.tsx` moved from `(tabs)/officials/committees.tsx` to a standalone `(tabs)/committees.tsx` on 2026-07-04 (route `/committees`, was `/officials/committees`) so it's its own top-level tab on mobile too, matching desktop's `WebTopNav` — it used to be a sub-tab buried inside Electeds, much less discoverable than on desktop. `officials/_layout.tsx`'s `SUBTABS` no longer lists it
- Detail screens at root level: `bill-detail`, `legislator-detail`, `committee-detail`, `quiz`
- `app/_layout.tsx` wraps everything in `AuthProvider` → `GamificationProvider` → `DesktopWebShell` (renders `WebTopNav` on desktop web, gated to `AUTHENTICATED_SEGMENTS` — `(tabs)`, `bill-detail`, `committee-detail`, `quiz` — so pre-auth routes like `/lookup` and `(auth)` keep only their own header). `(tabs)/_layout.tsx` branches: `DesktopLayout` (just a `<Slot/>`, nav lives in the shell) vs `MobileLayout` (bottom `Tabs`)
- `components/app-alert.tsx` (`AppAlert.alert`, `AppAlertHost` mounted in root layout) replaces RN's `Alert.alert` everywhere in the app — `react-native-web`'s `Alert.alert` is a no-op stub, so every call site used to silently do nothing on web. Delegates to the real native `Alert` off-web; use `AppAlert.alert`, never `Alert.alert`, in any new code
- Responsive logic: `hooks/use-responsive-layout.ts` — non-web is always "mobile"; web uses `Breakpoints` from `constants/theme.ts`; `showSidebar` only on desktop

## Data sources (`services/`)
- **Local-first legislator data**: `services/data/kansas-legislators.ts` (~4,400 lines) is **AUTO-GENERATED from KansasLegislators.swift — do not hand-edit**; regenerate with `node scripts/parse-kansas-legislators.js`. Contains chamber/district/party/email/phone/committees/leadership per legislator
- `services/kansas-legislators.ts`: adapter exposing the static data in OpenStates-shaped types. Synthetic IDs: legislators `ks-state-{chamber}-{district}`, committees `ks-committee-...` (check `isLocalLegislatorId`/`isLocalCommitteeId`)
- `services/openstates.ts`: OpenStates v3 API with a serialized fetch queue (1.1s min spacing, 60s rate-limit cooldown) + AsyncStorage persistent cache; falls back to/prefers local data for KS legislators. `getCommitteeAssignments` takes the full `Official` (not just an ID) and resolves non-local IDs (e.g. from `getOfficialsByLocation`'s address/geo lookup) to the local `ks-state-*` format by chamber+district before querying — officials found via location lookup have raw OpenStates IDs that never match local committee data otherwise
- `services/legiscan.ts`: LegiScan API for bills, sponsored bills (12h cache), voting records (24h cache), session list (12h cache), master list (30min cache; `forceRefresh` param bypasses for pull-to-refresh). Routes through the Netlify Function proxy: web uses the relative `/.netlify/functions/legiscan` path (so local web dev needs `npm run web:netlify`), native uses `EXPO_PUBLIC_LEGISCAN_PROXY_URL`; `EXPO_PUBLIC_LEGISCAN_API_KEY` is a dev-only direct-API fallback
- Bills screen hydrates committee names lazily via FlatList viewability (only visible cards trigger `getBill`), with a 7-day persisted name cache — do not reintroduce eager per-bill fetching; a full masterlist sweep is ~500 calls against a 30k/month quota
- `getBillDetail` (30min cache) and `getRollCall` (30-day cache, votes are immutable once cast) are cached globally by ID, shared across every caller — this matters because `fetchVotingRecord` scans the same ~200 recent bills for every legislator, so caching de-duplicates across legislators, not just within one. The scan itself runs on a 4-worker pool (`VOTE_SCAN_CONCURRENCY`), not sequentially
- `services/billtrack50.ts`: BillTrack50 scorecard (hardcoded `SCORECARD_ID = 4579`, "2026 KSLeg Scorecard")
- `services/legislator-match-engine.ts` + `bill-category-tagger.ts` + `hooks/use-quiz.ts`: quiz answers (1–5 per issue category) vs. legislator vote records → alignment/composite match scores
- `constants/committee-meetings.ts`: static meeting day/time/room, keyed `"${chamber} ${committeeName}"` where the name must **exactly match** the KANSAS_LEGISLATORS committee names
- `services/bill-committee.ts`: shared committee-name resolution (`resolveCommitteeName`, `findCommitteeActionFromHistory`, `COMMITTEE_UNAVAILABLE` = "No committee assigned yet"). Both `bills.tsx` and `bill-detail.tsx` use this so they never disagree about a bill's committee — `bill-detail.tsx` always renders the Committee row now (previously omitted it silently when unresolved, while the list showed a raw "Committee unavailable" string)
- `services/bill-status.ts`: shared `getBillStatusColor(status, fallback)` mapping a LegiScan status label to its badge color (Passed/Chaptered green, Introduced/Engrossed light-green, Enrolled teal, Vetoed/Failed red, else `fallback`). Consumed by `bills.tsx`, `bill-detail.tsx`, and `actions.tsx` — all three used to carry an identical private `getStatusColor` copy (2026-07-08: the bills LIST badge had silently dropped the color-coding and rendered flat gray while the detail/actions screens stayed colored; consolidated so they can't drift). Don't re-add a local status-color function to any bill-rendering screen; import this instead. Colors are the raw light-palette hex, not theme-aware — matches the pre-existing behavior of the detail screen

## Firebase / Firestore
- Config from `EXPO_PUBLIC_FIREBASE_*` env vars; `getAuth()` throws if `window` is undefined (SSR/static export guard)
- Collections in `firestore.rules`: `users` (+ subcollections `surveys`, `savedOfficials`, `quizs`), `openBills`, `committeeEmails`, `bills`, `testimonies`, `notes`, `surveys`, `survey_responses`, `billTalkingPoints/points`, `legislatorRatings` (doc ID format `userId_legislatorId`, rating 1–5 int enforced in rules), `legislatorRatingStats`. Default deny for everything else
- Roles: `user` / `admin` / `super_admin` stored on the user doc (role changes require super_admin; the old self-promotion rule was removed 2026-07-03)
- `contexts/auth-context.tsx` exposes `resetPassword(email)` (Firebase `sendPasswordResetEmail`), wired to a "Forgot password?" link on the login screen. Treats `auth/user-not-found` as success (same generic "check your email" message either way) to avoid leaking whether an email is registered — don't special-case that error to reveal account existence
- `testimonies/{id}`: written by `components/testimony-form.tsx` (only when the user is signed in) with contact info + submission `status` (`sent`/`saved`/`cancelled`/`undetermined`, straight from `MailComposer`'s result). Rules restrict read/write to the owner or admin — this collection existed in rules before 2026-07-04 but nothing wrote to it, and its `allow write` rule was actually broken for `create` (checked `resource.data.userId`, which doesn't exist on a new doc); fixed alongside wiring up the first real writer. On a genuine (non-`cancelled`) send the form now also records a `Contact Legislator` gamification action (no dedicated testimony action exists; award-on-attempt matches the legislator-detail contact flow) and pre-fills name/email from `useAuth` + address from `useUserProfile`
- `billTestimony/{billId}` (`.isOpen` bool): the admin per-bill open/close switch (`hooks/use-bill-testimony-status.ts`) and the Actions-tab "open for testimony" list (`hooks/use-open-testimony-bills.ts`). **This collection had NO rule in `firestore.rules` until 2026-07-08** — it hit the default-deny, so every status read failed (feature always showed "closed", Actions tab always empty) and the admin toggle write was rejected: the entire testimony feature was non-functional. Added `match /billTestimony/{billId} { allow read: if isAuthenticated(); allow write: if isAdmin(); }` (authed read is required at collection scope for both the single-doc listener and the `isOpen==true` list query). **The rule is in the file but NOT yet deployed** — it takes effect only after `firebase deploy --only firestore:rules` (see gotcha #2); until then production is still broken
- `legislatorRatings/{userId}_{legislatorId}` docs additionally store a `legislatorId` field (not required by the rules, but needed for `where('legislatorId', '==', ...)` queries) — `components/legislator-rating.tsx` reads/writes this to show a live average + let the signed-in user rate. No use of `legislatorRatingStats`; the average is computed client-side by querying all ratings for a legislator, not from a maintained aggregate

## Gamification
- `constants/gamification.ts`: 5 action types with points (Contact Legislator 50, Share 25, View Legislation 10, Quiz 200, Rate 15), levels, achievements. All 5 actions now have real UI behind them as of 2026-07-04 (Rate Legislator via `components/legislator-rating.tsx`, Share Content via `services/share.ts`)
- `contexts/gamification-context.tsx`: state + streaks + achievement unlocks, persisted to Firestore under the user; `AchievementCelebration` overlay rendered at root
- `components/legislator-rating.tsx`: 1-5 star rating widget shown on `legislator-detail.tsx`, writes to `legislatorRatings/{uid}_{legislatorId}` (existing collection/rules, unused before 2026-07-04). Average/count are computed by querying all ratings for a `legislatorId` client-side on each load/rate — no aggregate stats collection is maintained (`legislatorRatingStats` stays unused; would need a Cloud Function to keep in sync, out of scope for the Firebase Spark plan this project is on)
- `services/share.ts`: cross-platform `shareContent()` — native always works (real OS share sheet); web tries `navigator.share()` first, falls back to clipboard copy, and silently returns `'unavailable'` if both fail (e.g. no share API and clipboard permission denied) rather than showing a false-confidence success message
- Match score views (`components/legislator-match-detail.tsx`) intentionally look different depending on data source: BT50-scored legislators show one composite percentage (BillTrack50's API gives no per-category data), vote-record-scored legislators get a full per-issue breakdown. Both footers now say so explicitly (2026-07-04) — don't try to fabricate category data for BT50 matches, the source data genuinely doesn't support it
- Quiz completion (`app/quiz.tsx`) shows results before recording the `Quiz Completed` gamification action (`setShowResults(true)` fires, then `recordAction` is deferred ~700ms via `setTimeout`) — this exists specifically so an achievement-unlock modal doesn't render on top of the results screen before the user has seen their match; don't reorder this back

## Conventions
- Path alias `@/*` → repo root (tsconfig)
- Kebab-case file names; components in `components/` (themed-text/themed-view pattern from Expo template), hooks in `hooks/`, static data in `constants/` and `services/data/`
- Theme: `constants/theme.ts` (`Colors` light/dark palettes, `Breakpoints`); color scheme via `use-color-scheme` (+ `.web` variant)
- Header back buttons use `padding: Spacing.md` (12px) around a 24px icon → 48px touch target, meeting the 44px accessibility guideline (standardized across `bill-detail.tsx`, `committee-detail.tsx`, `quiz.tsx`, `legislator-detail.tsx` on 2026-07-04, previously `Spacing.xs`/4px ≈ 32px). Matching header spacers (`navSpacer`/`headerSpacer`, used to balance the title when there's no right-side action) are sized to 48px too — keep both in sync if either changes

## Gotchas / watch-outs
1. The old LegiScan API key (`18cf...e2a`) lives in git history — rotation pending as of 2026-07-03 (requires emailing LegiScan support); until then the exposed key is still live. New key goes in Netlify env + `.env` only
2. Firestore rules changes require `firebase deploy --only firestore:rules` — editing `firestore.rules` alone does nothing
3. `services/data/kansas-legislators.ts` is generated — edit the generator/source, not the file (exception: the "Veterans & Military Affairs" → "Veterans & Military" typo fix on 2026-07-03 was applied to the generated file directly; the upstream Swift source needs the same fix before regenerating)
4. Gamification points are client-trusted (any user can inflate their own via console) — matters if leaderboards are ever added. `bills`/`legislatorRatingStats` writes were locked to admin and `users/{uid}/actionHistory` + `billCategoryTags` rules added on 2026-07-03 (both were silently hitting default-deny before)
5. `npm run validate:data` checks `COMMITTEE_MEETINGS` keys against committees derived from the legislator data, and (added 2026-07-08) validates `constants/committee-emails.ts`: every key must match `^(Senate|House) Committee on .+$` (the `formatCommitteeName` shape used for the exact-key lookup) and every value must be a well-formed `@senate.ks.gov`/`@house.ks.gov` address — run it after touching any of those files (already in CI). NOTE: the email check only catches gross format errors, not a well-formed-but-wrong address. All 40 addresses were verified against the live Legislature site on 2026-07-09 (method below): 37 matched exactly, 1 was wrong and fixed (`Government Efficiency`: `kansascoge@senate.ks.gov` → `S.Govt.Efficiency@senate.ks.gov`), and 2 remain UNVERIFIABLE because the live committee pages publish no testimony email — `Senate Committee on Confirmation Oversight` (`kaleb.hagg@senate.ks.gov`, a personal mailbox; committee meets "On Call") and `Senate Committee on Local Government, Transparency and Ethics` (`transparency@senate.ks.gov`). Also MISSING from our map: `House Committee on Legislative Budget` (live `H.LegBudget@house.ks.gov`) — add only if bills actually get referred there. Verification method: the committee pages Cloudflare-obfuscate emails (`data-cfemail="<hex>"`), which WebFetch's summarizer both redacts and hallucinates — do NOT trust WebFetch for these. Instead `curl` the page and XOR-decode: first hex byte is the key, XOR each subsequent byte with it to get the address. The full standing-committee list comes from the htmx endpoints `/b2025_26/committees/?chamber=House|Senate&page=N&per_page=50` (send header `HX-Request: true`), since the JS index truncates
6. Verify changes with `npx tsc --noEmit` + `npx expo lint` + `npm test` + `npm run validate:data` — all four are clean as of 2026-07-03 and enforced by CI. The `@ts-expect-error` on `getReactNativePersistence` in `services/firebase.ts` covers a firebase v12 typings gap; remove it if a firebase upgrade makes tsc complain
7. The Netlify proxy's rate limit is in-memory per function instance — real protection depends on Netlify/Lambda reusing a warm instance across a burst, which `netlify dev` locally does *not* do (confirmed: each local invocation gets a fresh module scope, so the counter never accumulates). It stops a naive sequential hammering script; it does not stop a slow, patient, or highly-parallelized attack — that needs a persistent store (e.g. Netlify Blobs)
8. Gamification's `users/{uid}` `totalPoints` write is now bounded by `isValidPointsWrite()` in `firestore.rules` (2026-07-04): never decreases, and a single write can't increase it by more than 200 (the largest real action, Quiz Completed — see `constants/gamification.ts` `ACTION_POINTS`). This is deliberately a range check, not an exact-match-per-action check — the latter would reject legitimate writes if a user's local gamification state drifts from Firestore's current value (e.g. the app open on two devices at once), and there's still no rules emulator/test rig in this repo to safely verify anything stricter. Verified directly against production via the Firestore REST API with a real ID token (bypassing the app entirely): a 999999999 jump, a +201 jump, and any decrease are all rejected with 403 `PERMISSION_DENIED`; a +200 jump succeeds. **`unlockedAchievements` and `actionCounts` remain completely unvalidated** — a client can still set `unlockedAchievements` to any array of achievement ids directly, deliberately left out of scope (doing it properly means duplicating `ALL_ACHIEVEMENTS`' thresholds into rules, which will drift from `constants/gamification.ts` over time). The only way to close that gap correctly is moving point/achievement awarding to a Cloud Function (requires switching the Firebase project off the free Spark plan to Blaze)
9. `components/ui/icon-symbol.tsx`'s `MAPPING` must list every SF Symbol name ever passed to `IconSymbol` — `icon-symbol.ios.tsx` accepts any name (real SF Symbols), so a missing entry only breaks web/Android, and silently: unmapped names render as blank space, not an error, not even a `?` placeholder. Confirmed and fixed 6 missing entries on 2026-07-03 (`chevron.left`, `arrow.up.right`, `calendar`, `clock`, `mappin`, `doc.text.magnifyingglass`) that had been invisible back-buttons/icons app-wide on non-iOS
10. Testimony content deliberately excludes email/street address/state/zip — only first/last name + city go into the testimony sent to committees (confirmed intentional 2026-07-04). Those fields are still collected in the form and now saved to Firestore alongside the submission for the user's own record, just never included in what's sent to the committee. When a bill's committee has no email on file, the form links out to the official Kansas Legislature committee directory (`kslegislature.gov/li/b2025_26/committees/`, verified live 2026-07-04) instead of a vague "submit manually" — that URL is session-specific (`b2025_26`) and will need updating whenever a new legislative session starts
11. `expo-mail-composer`'s `composeAsync` result is only trustworthy on iOS. On web it always returns `status: 'undetermined'` (it just does `window.open('mailto:...')` with no way to detect if the user actually hit send — confirmed by reading `ExpoMailComposer.web.ts`); per the library's own doc comment, Android also always reports `'sent'` regardless of what really happened. `components/testimony-form.tsx`'s `STATUS_MESSAGES` word the `undetermined` case as genuinely uncertain rather than claiming false success — don't "fix" that wording to claim more certainty than the platform can actually provide
12. `points.tsx` is intentionally not a tab-bar item (`href: null` in `(tabs)/_layout.tsx`) — it's reached via the "View Details" button on Profile's level/points card (`router.push('/(tabs)/points')`), not a persistent nav slot. Profile's card used to duplicate achievements/ways-to-earn/recent-activity inline (removed 2026-07-04); `points.tsx` is now the single source of truth for all of that

This is a browser-automation limitation, not a code bug; the unit tests in `services/__tests__/share.test.ts` mock `Share.share`/clipboard directly instead of relying on a live browser
